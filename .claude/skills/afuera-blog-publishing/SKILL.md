# Afuera Blog Publishing

This skill encodes the end-to-end pipeline for publishing a blog post on [afuerai.com](https://www.afuerai.com). The site is a static build: markdown in `/content/blog/*.md` → `/blog/<slug>/index.html` via `scripts/build.mjs`, deployed by Vercel on push to `main`.

**Key principle:** this pipeline is already built and working. Do not reinvent steps. Follow the checklist, fix what the lint flags, and commit with a clear message.

## 0. Repo layout

```
afuera-site/
├── content/blog/                  # SOURCE: one .md per post (tracked)
├── scripts/
│   ├── build.mjs                  # markdown → static HTML
│   ├── templates.mjs              # shared <head>, nav, footer, CSS
│   └── generate-card.py           # Python + Pillow card generator
├── images/blog/                   # post hero images (WebP, tracked)
├── blog/                          # BUILD OUTPUT (.gitignored)
├── author/                        # BUILD OUTPUT (.gitignored)
└── index.html                     # landing page (build injects blog preview)
```

Never edit anything in `/blog/` or `/author/` directly — they are regenerated by `npm run build`.

## 1. Intake

What you'll receive from the user, in some combination:
- A markdown file (inline paste, or a path to drop in).
- Optionally a source hero image (PNG, JPG, or a prompt to generate one).
- Sometimes just a title + topic + cluster and a "draft and publish it."

First thing: **confirm the slug.** Slug drives the URL (`/blog/<slug>/`), the image filename (`<slug>-card.webp`), and the markdown filename (`<slug>.md`). If the user didn't provide one, derive from the title, run it by them before writing files.

## 2. Frontmatter schema

Every post needs this frontmatter block. The build accepts both my native schema (`seoTitle`, `summary`, `date`, `cover`) and the content-pipeline schema (`seo_title`, `description`, `date_published`, `featured_image`). Prefer the content-pipeline names — they're what the user's drafts arrive in.

```yaml
---
title: "Voice title — shown as the H1 and on cards"
seo_title: "SEO title — goes in <title>, schema.headline, and renders as the H2 dek under H1"
slug: "url-safe-slug"
description: "One to three sentences. Used as summary, meta description, OG/Twitter description, and TL;DR block."
date_published: "2026-04-24"
date_modified: "2026-04-24"
author: "Ali Murphy"
cluster: "Field Operations"   # see §3 — drives card pattern + eyebrow color
role: "AEO-forward"           # optional taxonomy
icps: ["GTO", "EC", "DMC", "TA", "TD", "LD"]   # see §4
angle: "AI Strategy"          # optional
format: "Listicle"            # optional
funnel: "TOFU"                # optional
tags: ["AI readiness", "AI strategy", "travel operators"]
featured_image: "/images/blog/<slug>-card.webp"
featured_image_alt: "Short descriptive alt text — what's actually in the image"
faq:
  - question: "Short natural question a reader would ask?"
    answer: "Direct answer in one paragraph. Plain voice. No em dashes."
  - question: "..."
    answer: "..."
---
```

**Voice title vs SEO title.** Voice title is the editorial hook (short, punchy, human). SEO title is the keyword-loaded one used in `<title>`, `schema.org` headline, and rendered visibly as an H2 dek under the H1. They should be different. If the user gives you only one, ask which it is.

**FAQ block.** 5–7 questions minimum. Each answer is a full sentence or two (used in FAQPage JSON-LD for AI Overviews + ChatGPT). Questions should be phrased the way someone would actually type them, not marketing-ese.

## 3. Cluster taxonomy

The `cluster` field drives the card-image color pattern (`scripts/generate-card.py`) and the orange eyebrow label on the card + post page. Allowed values:

| Cluster | Card pattern | Use for |
|---|---|---|
| `Field Operations` | teal waves | Guides, on-trip ops, guest-on-the-ground |
| `Office Operations` | orange grid | Back-office, proposals, accounting, CRM |
| `Marketing` | pink dots | Content, SEO/AEO, brand, website |
| `AI Strategy` | yellow diagonals | How-to-think-about-AI posts, readiness, audits |
| `Industry POV` | orange grid | Opinion, commentary on the travel industry |

If the user picks a new cluster name, either map it to one of the above or add a new pattern to `generate-card.py`.

## 4. ICP codes

The `icps` array is the audience tag. Build expands codes to full labels on the card and filter chips. Known codes:

| Code | Label |
|---|---|
| `GTO` | Global Tour Operator |
| `ITL` | Independent Trip Leader |
| `TA` | Travel Advisor |
| `TD` | Trip Designer |
| `EC` | Experience Company |
| `DMC` | DMC |
| `LD` | Luxury Trip Designer |

Don't invent new codes without updating `ICP_LABELS` in `build.mjs`.

## 5. Hero image — two branches

Every post needs a 1200×800 WebP at `images/blog/<slug>-card.webp`. This image is used three places: card in the hub, hero image on the post page itself, and OG/Twitter share. Two ways to get it:

### Branch A: user provides a source image

They'll hand you a PNG/JPG (often from `~/Desktop/afuera-site/Afuera blogs/` or similar — that folder is `.gitignored`). Compress to 1200×800 WebP q82:

```bash
cd ~/Desktop/afuera-site && python3 -c "
from PIL import Image
src = Image.open('PATH/TO/source.png')
src.thumbnail((1200, 800), Image.LANCZOS)
# Or for hard 3:2 crop: src = src.resize((1200, 800), Image.LANCZOS) after cropping
src.convert('RGB').save('images/blog/<slug>-card.webp', 'WEBP', quality=82, method=6)
"
```

Expect ~90–95% size reduction from source. If the output is above 300 KB something's off.

### Branch B: generate a branded card

Use the Python generator. It produces a cluster-patterned card with the title and cluster label baked in.

```bash
cd ~/Desktop/afuera-site && python3 scripts/generate-card.py \
  --title "The voice title here" \
  --cluster "Field Operations" \
  --slug "<slug>"
```

Output lands at `images/blog/<slug>-card.png`. Convert to WebP immediately (same PIL one-liner as above) and delete the PNG. The final `featured_image` frontmatter value is `/images/blog/<slug>-card.webp`.

**Do not retroactively regenerate cards for existing posts.** Only generate the card for the post you're currently publishing. The user corrected this explicitly once — don't repeat the mistake.

## 6. Place the markdown

Write the markdown to `content/blog/<slug>.md`. Use the exact `slug` from frontmatter.

Before writing, sanity-check:
- H1 is not in the body — the build renders the H1 from frontmatter `title`.
- First visible text is a prose lede paragraph, not an intro H2.
- Heading hierarchy: H2 for top-level sections (these become the TOC), H3 for subsections. Never skip a level.
- Tables use GFM pipe syntax. Build wraps them in `.table-wrap` automatically.
- Frontmatter quotes use straight double-quotes. Curly quotes in YAML will break `gray-matter`.

## 7. Build

```bash
cd ~/Desktop/afuera-site && npm run build
```

Expect three kinds of output:
1. `[afuera]` lines listing each post and the author page written.
2. `[voice]` warnings per post — lint results. Do not ignore entirely; see §8.
3. The summary line at the bottom.

Build time is <2 seconds. If it throws, 9/10 times the cause is a frontmatter YAML error — check curly quotes, unescaped colons in titles, or unmatched brackets in `icps`.

## 8. Voice lint — review, fix the real ones

The build runs `lintVoicePost()` and logs warnings for:
- **Em dashes** (`—`, `–`) — the Afuera voice uses plain prose. Replace with a period, comma, or rewrite. These are the most common warning and usually the most worth fixing.
- **Banned buzzwords** — `leverage`, `unlock`, `streamline`, `transform`, `revolutionize`, `seamless`, `cutting-edge`, `world-class`, `best-in-class`, `next-generation`, `synergy`, `game-changer`, `robust`, `holistic`, `paradigm`. All should be rewritten. No exceptions. *"Leverage"* especially — always replace.
- **Hedges** — `somewhat`, `perhaps`, `potentially`, `arguably`, `seemingly`. Often the sentence is stronger without them. Judgment call.
- **Weak adverbs** — `really`, `very`, `quite`, `extremely`, `significantly`, `incredibly`. Usually deletable.
- **Generic transitions** — `"here's the kicker"`, `"the thing is"`, `"at the end of the day"`, `"let me explain"`. Cut or rewrite.

**Process when reviewing warnings:**
1. Triage: em dashes + banned words first (objective fixes), then hedges/adverbs/transitions (stylistic).
2. For each fix, read the full line in context — a flagged word in a quote or in a table is often fine; use judgment.
3. Edit the source markdown (`content/blog/<slug>.md`), then re-run `npm run build`.
4. Confirm warnings drop, then move on. Don't chase zero — the point is awareness, not gatekeeping.

For the AI-readiness + older posts we already know have warnings, only fix warnings on the post you're currently publishing. Don't refactor other people's posts to hit zero.

## 9. Local preview (optional)

If the user asks to see it live before push:

```bash
cd ~/Desktop/afuera-site && python3 -m http.server 8000
```

Open `http://localhost:8000/blog/<slug>/` and the homepage at `http://localhost:8000/` to see the preview card injected.

## 10. Commit + deploy

Only commit tracked files. `/blog/`, `/author/`, `/sitemap.xml`, `/robots.txt`, `/llms.txt`, and `/privacy/` + `/terms/` are `.gitignored` — they rebuild on Vercel.

Tracked artifacts you should add for a new post:
- `content/blog/<slug>.md`
- `images/blog/<slug>-card.webp`
- Sometimes `index.html` (the build updates the homepage blog-preview block; the diff is usually ±3 post cards)

**Do not add:**
- Anything under `/blog/` or `/author/` (ignored, rebuilt)
- `.DS_Store`, `Afuera blogs/` source images (ignored)
- Source PNGs from `generate-card.py` — only ship the `.webp`

### Commit message style

Match the existing repo style: imperative, 60-char-ish subject, optional body. Examples from history:

- `Publish "AI Readiness Assessment" + fix TOC double-encoding`
- `Editorial table styling — tables now read as designed objects`
- `Render seo_title as a visible H2 subtitle under every post H1`

For a routine publish: `Publish "<voice title>"`. If you also bug-fixed or refactored something, list it after a plus or on a second line.

```bash
cd ~/Desktop/afuera-site && \
git add content/blog/<slug>.md images/blog/<slug>-card.webp index.html && \
git commit -m "$(cat <<'EOF'
Publish "<voice title>"

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
EOF
)"
```

### Deploy

```bash
git push origin main
```

Vercel rebuilds on push. Deploy time is ~15 seconds.

### Post-deploy verification

After the user says "deploy" or after pushing, confirm these are live:
- `https://www.afuerai.com/blog/<slug>/` — post renders with hero image, byline, TL;DR, TOC (if ≥2 H2s), FAQ, CTA.
- `https://www.afuerai.com/blog/` — new card appears at the top.
- `https://www.afuerai.com/` — homepage preview shows the new post in the 3-card strip.
- `https://www.afuerai.com/sitemap.xml` — new URL listed with today's `lastmod`.

If any of those are off, the build probably succeeded locally but the git push didn't include a tracked file (commonly the `.webp`).

## 11. Updating an existing post

Same pipeline, but:
1. Edit `content/blog/<slug>.md` and bump `date_modified` to today.
2. `npm run build` — the sitemap `lastmod` will reflect the new `date_modified`, and the post page's "Updated" line will show a relative date.
3. Commit: `Update "<voice title>": <one-line reason>`.

Do not bump `date_published` on an edit — that's what Google and AI crawlers use to decide "is this article new." Only `date_modified` moves.

## 12. Anti-patterns — do not do these

- **Editing `/blog/<slug>/index.html` directly.** It's regenerated on every build. Edit the markdown.
- **Generating cards for posts other than the one being published.** Only the current post.
- **Batch-fixing voice warnings across all old posts.** Only the current post. Drift is fine.
- **Adding new fields to frontmatter without updating `readPosts()` in `build.mjs`.** The parser only picks up fields it knows about.
- **Using curly quotes in YAML frontmatter.** `gray-matter` will choke.
- **Committing `/blog/`, `/author/`, `/sitemap.xml`, `/robots.txt`, `/llms.txt`.** Build output is gitignored. If git status shows any of these, something is wrong.
- **Skipping the hero image.** The build works without it (falls back to `/hero-graphic.png`), but the card and post page look empty. Always ship with a `-card.webp`.
- **Pushing without `npm run build` succeeding locally first.** Vercel runs the same build; if it fails there, the deploy errors and the post won't appear.

## 13. Quick-reference publish loop

For the common case — user hands you a markdown + source image, slug already decided:

```
1. Compress source image → images/blog/<slug>-card.webp (PIL, 1200×800, q82)
2. Write markdown → content/blog/<slug>.md
3. npm run build
4. Review [voice] warnings, fix em dashes + banned words in the new post only
5. npm run build (again, confirm warnings dropped)
6. git add content/blog/<slug>.md images/blog/<slug>-card.webp index.html
7. git commit -m 'Publish "<voice title>"'
8. git push origin main
9. (wait ~15s) verify /blog/<slug>/ is live
```

That's it. Seven to nine steps, ~5 minutes from clean markdown to live post.
